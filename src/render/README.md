# src › render

Rendering pipeline and ray-tracing engine.

- `integrators/` – Dedicated modules for the renderer’s sampling strategies.
  - `integrator_common.c` – Owns the `IntegratorContext`, tile allocator/clear helpers, RNG seeding, and the `IrradianceCache` container used by the camera pass. Any integrator that opts into tile-based accumulation or cache population goes through these helpers so lifetime/clearing stays consistent.
  - `forward_light_integrator.c` – Emits BSDF-driven light paths using the same Lambert/GGX materials as the camera pass, accumulating radiance in the same units before optional diffusion and tonemapping.
  - `camera_path_integrator.c` – Hybrid path tracer split into three stages (orthographic top-down view; every pixel shares the same view direction):
    1. **Direct pass** – Every pixel is converted to world-space through the shared `Camera`, checked for line-of-sight to the light using uniform-grid shadow rays, and (when visible) assigned the unified inverse-square falloff seeded with the light’s disk radius/area. Radiance is written additively via tile grids or the linear buffer so indirect light can layer on top naturally before the higher exposure tonemap kicks in.
    2. **Indirect pass** – For pixels that stayed black after the direct sweep, the integrator fires 48 jittered “feeler” rays per pixel. Each hit point looks up its owning `SurfaceSegment` (with circle hits falling back to their analytic normals), checks that the segment faces the pixel, that the light strikes the sampled spot, and that the segment can see the pixel (all resolved with uniform-grid occlusion tests). Incoming energy is estimated from the irradiance cache if available (otherwise falls back to `ComputeDirectRadiance` at the hit point), then every valid contribution is tempered by the shared inverse-square + exponential decay, a sensor-facing term, and per-sample/per-pixel clamps before being *summed* through the additive accumulator so multiple surfaces can light the same sensor without fighting.
    3. **Tonemap pass** – Converts accumulated energy to 8-bit output using an exponential exposure curve (exposure 0.2, gamma 0.55). When tiles are enabled each tile tonemaps independently before writing into the shared pixel buffer.
  - `camera_path_integrator_disney.c` – Experimental 2.5D MIS path tracer (paused in the UI while the renderer is kept 2D).
- `ray_tracing2.c` – Orchestrates renderer lifecycle, resource ownership, blur post-process, and delegates to the selected integrator.
- `render_helper.c` – SDL drawing helpers for circles, polygons, UI text, and light-strength calculations shared by the renderer and editors.
- `uniform_grid.c` – Builds the reusable grid accelerator backed by per-cell object lists. Bounds expand around the current scene, every object splats its AABB into the grid, and traversal uses a 2D-DDA to test only geometry inside each traversed cell. Every ray (camera feelers, light shadow checks, cache occlusion probes) shares the same `UniformGridTraceRay` so hit filtering and epsilon offsets stay identical between subsystems.
- `material_bsdf.c` – Converts scene objects into compact material descriptors and implements shared Lambert/GGX BSDF evaluation utilities used by both integrators.
- `runtime_material_texture_3d.c` – App-local procedural material texture sampler for native `3D` hits and Material preview. It supports triangle-barycentric UV anchoring plus explicit placed-UV sampling for editor face islands, with rust/fog overlays controlled by a node-ready parameter block: pattern mode, coverage, grain, edge softness, contrast, flow, color depth, surface damage, and seed.
- `runtime_material_texture_stack_3d.c` – Procedural material stack evaluator for base/overlay layers. It resolves stack color and physical response into `RuntimeMaterialSurfaceEval` and exposes procedural physical-channel readback for roughness, reflectivity compatibility, specular weight, and diffuse response without promoting layer masks or transparency into physical opacity/transmission.
- `runtime_material_graph_3d.c` / `runtime_material_graph_3d_json.c` – Node-ready material graph compiler adapter and graph JSON serializer. It accepts a bounded graph document with durable node ids, compiles layer nodes into `RuntimeMaterialTextureStack`, records supported M6 channel-output refs beside the stack, persists camelCase/snake_case graph documents for scene-config/runtime-scene compatibility, and rejects unsupported/deferred outputs such as `displacement.height` without changing renderer sampling or Material editor UI.
- `runtime_material_payload_3d.c` – Resolves scene-object material presets into the native `3D` shading payload. Runtime hits now carry a per-object local triangle ordinal so generated Material face-group texture overrides can be sampled as cohesive face islands and applied to actual BSDF roughness, reflectivity/specular weight, diffuse response, color, and transparency; color depth and surface damage control how strongly the texture changes color and material response. Glass objects can carry object-local transport overrides for transmission, optical IOR, absorption distance, and thin-walled behavior while preserving the global transparent preset as the default seed. Explicit glass transport overrides decouple direct-light shadow transmittance from low visual alpha, so lens fixtures can render as faint glass without turning physical visibility into an opaque shadow. Mirror objects can carry object-local response overrides for reflectivity, roughness/sharpness, specular strength, and tint while preserving the global Mirror preset as the default seed.
- `runtime_material_authored_texture_3d.c` / `runtime_material_authored_texture_3d_manifest.c` – Authored texture binding and manifest loader for native `3D` material payloads. Manifest surfaces can carry bounded explicit channel references such as `base_color.rgb` and `roughness.scalar`; the loader records those refs for readback without changing sampling, payload conversion, or BSDF semantics. Normal and bump channel names classify only as future shading-normal inputs, and `displacement.height` remains an explicit deferred channel name that is not accepted by manifest binding.
- `space_mode_adapter.c` – Central mode adapter seam for world/screen conversion dispatch plus ray/hit setup helpers. Current `SPACE_MODE_3D` path preserves 2D behavior intentionally until later parity slices.
- `ray_tracing_mode_backend.c` – Backend route resolver for runtime mode dispatch. It keeps `2D` as canonical, preserves compat-fallback `3D` for unsupported/incomplete scenes, and now activates bounded native `3D` only when retained seeds can build a runtime scene with triangles plus authored light/camera samples.
- `runtime_camera_3d_rays.c` – Native `3D` camera projector for the runtime lane. It builds a bounded projector from `RuntimeCamera3D`, emits proof-scene camera rays, and projects runtime world points back to screen space so live light markers can use the same camera contract as native shading.
- `runtime_direct_light_3d.c` – Native direct-light shading rule for the runtime lane. It shades first-hit camera rays using a renderer `RuntimeLightSet3D` accumulator, preserves the single `RuntimeScene3D.light` compatibility wrapper, and can sum deterministic RGB contribution across multiple enabled finite lights. Each light still uses the native 3D visibility helpers plus the 16-slot hit-seeded stratified finite-radius sample population; area-light shadow checks start with 4 samples and stop there for clearly fully visible or fully blocked windows, escalating to 8 samples only for mixed, penumbra, or partial-transparency cases. Result diagnostics report evaluated lights, contributing lights, visibility rays, and peak per-light contribution.
- `runtime_light_set_3d.c` – Native `3D` light registry contract. It owns renderer-local light source storage, RGB color and shape metadata, enabled-light counting/lookup, deep copy/reset/free behavior, source-origin removal, and compatibility seeding from the current single `RuntimeLight3D` field. Direct-light shading consumes only enabled entries through explicit light-set entrypoints, runtime-scene authored `lights[]` build into the scene-owned light set, simple material emitters can register as enabled static sphere proxies, flat material panels can register as enabled static `rect` proxies, and complex mesh material emitters stay disabled `mesh_emissive` diagnostics with aggregate area/weight/centroid/normal/proxy-radius stats plus a `meshAreaSamplerOnly` flag. Until per-light motion paths exist, the first enabled registered light is the sole Bezier-driven moving light; later registered lights remain static.
- `runtime_diffuse_bounce_3d.c` – Bounded native diffuse-bounce tier for the runtime lane. It owns the dedicated `Diffuse Bounce` renderer branch, reuses the shared primary-hit/direct-light seam, and now adds a fixed-sample secondary scene-response pass so shadowed first-hit surfaces can receive limited indirect lift.
- `runtime_emissive_light_set_3d.c` – Cached native `3D` emissive-light candidate set. It scans runtime triangles during scene capability refresh, collects resolved emissive material triangles with centroid/area/weight metadata, stores primitive metadata for plane emitters, stores a compact cumulative distribution so Disney v2 area-light sampling does not scan the full mesh per receiver hit, and bridges material emitters into the registry: single-candidate/simple emitters become enabled static sphere proxies, flat same-plane panel emitters become enabled static `rect` proxies, and arbitrary mesh emitters remain disabled `mesh_emissive` diagnostics with object-level aggregate stats while the cached mesh-area sampler remains the active complex-emitter lighting path.
- `runtime_emissive_direct_3d.c` – Native emissive-material area-light sampler. It samples visible area contribution from the current receiver hit through the cached emissive-light candidate set, returns RGB contribution plus sampled/contributing diagnostics, and reports candidate count, selected candidates, visibility rays, selected sample geometry, candidate-selection PDF, area PDF, solid-angle light PDF, and any invalid-cache full-scan fallback. When derived scene capabilities prove emission support can be skipped, it takes the no-emissive fast path instead of sampling mesh emitters.
- `runtime_disney_v2_3d.c` – Experimental Disney v2 native `3D` route for the principled material lane. It consumes cached native `3D` payloads through `RuntimePrincipledBSDF3D`, exposes deterministic lobe/direct/emission/transmission diagnostics plus a route-proof diagnostic summary, and is selectable through the distinct `disney_v2` integrator identity without replacing the shipped Disney combiner. The first stochastic slice records a one-bounce path state with sampled diffuse/specular/transmission lobe, support ray, RGB throughput, BSDF/light PDFs, power-heuristic MIS weights, stochastic direct/BSDF radiance channels, secondary ray/hit counters, explicit light-emitter hit state for finite-radius runtime light proofing, and a resolved secondary-hit payload/principled vertex diagnostic that modulates first secondary contribution. Disney v2 now also records per-vertex MIS accounting for light-sampled versus BSDF-sampled contribution branches, finite-light versus emissive-material emitter classification, aggregate branch contributions, branch-removed radiance diagnostics, ordinary emissive-material endpoint contribution from BSDF support rays, primary-vertex emissive-area contribution from cached ordinary emissive-triangle candidates, emissive-area candidate/visibility/fallback diagnostics, branch-separated finite-light versus emissive-area direct-light MIS records, and a bounded mirror/glossy reflection channel that reuses native specular reflection tracing for crisp primary reflection while handing reflected geometry recursion to the Disney v2 transport sidecar.
- `runtime_caustic_settings_3d.c` – Phase 0/PPM-0A spatial-caustic contract owner. It keeps renderer-level caustic state separate from the Disney v2 analytic sidecar: `analytic` remains the receiver-hit proof/reference mode, `spatial_cache` is accepted as an inspectable no-op contract state, and `transport` is the explicit caustic transport request mode. Headless summaries report requested volume/surface cache state, selected transport engine, photon-map contract/readback state, and the no-sampleable-VF3D volume suppression state.
